# 声网（Agora）RTC 行业供需、UDP 优化与技术护城河

日期：2026-06-06

## 核心结论

声网所处的高端 RTC 领域，需求正在从传统“人和人音视频通话”升级为“人-人、人-AI、多模态、低延迟、高并发实时交互”。基础 RTC 能力正在商品化，但高端 RTC 仍有门槛，尤其是在弱网、多方互动、跨地域、低延迟、高并发、实时质量诊断和 AI 语音交互场景中。

声网技术中最有护城河的不是 UDP 协议本身，也不是普通 RTC SDK，而是：

> SD-RTN 实时传输网络 + 弱网 QoE 算法 + 多端 SDK 工程 + 实时质量诊断与运维体系。

这套能力在普通音视频场景中会被 WebRTC、开源方案和云厂商压价；但在高端实时互动、游戏语音、语聊房、跨国社交、在线教育、金融视频客服、智能硬件和 AI 语音 Agent 中，仍然有较强价值。

---

## 一、声网所处 RTC 高端领域的供需变化

### 1. 需求端：从“音视频通话”升级为“实时互动基础设施”

高端 RTC 需求不再只是普通视频会议或一对一音视频通话，而是扩展到更多实时互动场景：

- 社交娱乐、语聊房、游戏语音；
- 直播连麦、直播电商；
- 在线教育、远程医疗；
- 金融视频客服、双录；
- 智能硬件、IoT；
- AI 语音客服、AI 陪练、AI 教师、AI Avatar、AI 伴侣。

传统音视频通话需求仍在，但增长质量分化。普通一对一通话、简单会议、简单直播等技术门槛下降明显；真正有价值的是复杂场景下的低延迟、稳定性、弱网体验和多端适配。

### 2. 高端需求的定义变化：从“多人”变成“多并发 + 高体验一致性”

过去容易把高端 RTC 理解为“一个房间里很多人同时连麦”。现在更准确的定义是：大量实时会话同时发生，并且每个会话都要求低延迟、抗弱网、低抖动、低丢包、稳定连接和可诊断。

例如：

- AI 客服可能是大量一对一并发语音会话；
- 游戏语音和语聊房是中小房间高频互动；
- 直播电商是大规模观众 + 少量实时连麦；
- 在线教育和金融视频客服要求低故障率、低延迟和合规留痕。

因此，行业需求不是简单追求“更多人同时在一个房间里说话”，而是追求：

> 大规模并发条件下，每个实时会话都保持稳定、低延迟、可交互、可监控。

### 3. AI Agent 制造新的 RTC 需求

AI 语音助手、AI 客服、AI 陪练、AI 教师、AI Avatar 等应用，本质上需要“实时语音交互基础设施”。它们对 RTC 的要求比传统通话更高：

- 响应延迟要低，否则不像自然对话；
- 要支持打断，用户讲话时 AI 需要及时停止输出；
- 要抗噪声和回声；
- 要适配手机、网页、电话、智能硬件等多端；
- 要能串联 ASR、LLM、TTS、音频处理、Avatar、PSTN/SIP 等组件。

声网在 AI 语音交互中的机会，不是“接入 LLM”本身，而是当 AI Agent 进入真实语音环境时，低延迟、弱网、噪声、回声、端侧兼容和大规模并发都会重新成为门槛。

## 二、供给端：基础 RTC 过剩，高端 RTC 仍稀缺

### 1. 基础 RTC 供给明显增加

WebRTC 已经成为开放标准，浏览器和主流客户端都可以实现实时音频、视频和数据传输。开源 SFU、LiveKit、Daily、Twilio、腾讯云、火山引擎、网易云信等都能提供不同层次的实时音视频能力。

这意味着基础 RTC 的进入门槛明显下降。客户如果只是需要“能通话”，选择很多，价格会越来越透明。

因此，低端 RTC 会持续商品化：

- 普通一对一音视频；
- 简单客服通话；
- 普通会议；
- 内部工具；
- 对弱网和低延迟不敏感的场景。

### 2. 高端 RTC 供给仍然稀缺

高端 RTC 的难点不是“能不能连上”，而是：

- 弱网下是否不断线；
- 跨国链路是否低延迟；
- 大量并发会话是否稳定；
- 多人连麦是否低抖动；
- 移动端切换 Wi-Fi/5G 是否平滑；
- Android 碎片化设备是否兼容；
- 直播间或语聊房爆量时能否自动扩容；
- 出问题后能否定位到地区、运营商、机型、SDK 版本、链路节点。

这些能力需要全球节点、路由调度、端侧 SDK、音视频算法、真实网络数据和运维体系长期积累。不是单靠 WebRTC Demo 或开源 SFU 就能稳定实现。

## 三、声网在该细分领域的竞争力

声网的核心竞争力不是单点技术，而是一整套实时互动基础设施。

### 1. 全球实时网络与弱网优化

声网的核心技术资产是 SD-RTN，即 Software-Defined Real-Time Network。可以理解为覆盖在公网之上的实时传输网络。

其价值在于：

- 将用户流量接入距离更近的边缘节点；
- 根据实时网络质量选择更优路径；
- 在公网拥堵或跨境链路不佳时绕开差路径；
- 通过冗余传输、实时调度、丢包恢复等方法降低卡顿；
- 为高端客户提供稳定性和可诊断性。

这部分能力最适合：

- 跨国社交；
- 海外直播；
- 东南亚、印度、拉美等弱网地区；
- 全球 AI Agent 服务；
- 游戏语音；
- 高并发直播互动。

### 2. 多场景产品深度

声网不是只提供音视频 SDK，而是覆盖：

- 语音通话；
- 视频通话；
- 互动直播；
- 实时消息；
- 信令；
- 实时转写；
- AI 降噪；
- 云端录制；
- 质量分析；
- Conversational AI Engine。

这种产品深度使它更适合复杂场景，而不是只适合普通视频会议。

### 3. AI 语音方向的产品匹配度较高

AI 语音交互最怕三件事：

- 延迟高；
- 不能自然打断；
- 噪声环境下识别差。

声网原有 RTC 能力正好对应这些痛点：低延迟传输、音频处理、弱网优化、多端适配、实时质量监控。其 Conversational AI Engine 试图把这些能力延伸到 AI 语音 Agent 场景。

需要注意的是，AI Agent 编排层本身不一定是声网最强护城河，因为 LiveKit、Twilio 等也在做类似方向。声网真正的机会在于：当 AI Agent 从文本聊天进入真实语音、嘈杂环境、弱网国家、智能硬件和大规模并发场景时，它原有的 RTC 工程积累能否显著降低延迟和失败率。

## 四、UDP 优化是什么？

### 1. UDP 与 TCP 的区别

UDP 可以理解为：

> 尽快把包发出去，但不保证一定送达、不保证顺序、不自动重传。

TCP 可以理解为：

> 保证可靠送达，但一旦丢包会等待重传，容易造成延迟和抖动。

实时音视频更怕“迟到”，不一定怕“丢一点”。例如语音通话中，一个 20ms 的音频包如果迟到 500ms，即使完整到达也已经没有意义，不如直接丢掉，继续播放后面的声音。

因此，RTC 通常更偏向 UDP。

### 2. UDP 优化主要优化什么？

#### 2.1 抗丢包

UDP 丢包后不会自动重传，所以 RTC 厂商要自己处理。

常见方法包括：

- **FEC 前向纠错**：发送额外冗余数据，接收端可用冗余包恢复部分丢失数据；
- **选择性重传**：判断丢失包是否还来得及补救，来得及才重传；
- **丢包隐藏 PLC**：语音包丢失后，用算法估计缺失片段，使听感尽量连续。

#### 2.2 抗抖动

网络包到达时间并不稳定。RTC 需要动态抖动缓冲区，即 jitter buffer。

难点在于：

- 缓冲太短，容易卡顿；
- 缓冲太长，延迟变高；
- 缓冲需要根据网络状态实时调整。

高端 RTC 追求的不是单纯“平均延迟低”，而是：

> 低延迟下仍然稳定不卡。

#### 2.3 拥塞控制

UDP 没有天然成熟的拥塞控制机制，RTC 厂商需要自己判断网络是否拥堵，并动态调整码率。

例如：

- 网络好时提高视频清晰度；
- 网络差时降低分辨率、帧率和码率；
- 语音优先于视频；
- 屏幕共享优先于摄像头；
- 关键帧优先于普通帧。

这类能力也叫自适应码率、拥塞控制、QoS 调度。

#### 2.4 多路径与路由优化

UDP 包可以直接走公网，但公网路由不一定最优。高端 RTC 厂商通常会建设实时传输网络，让用户接入最近节点，再通过优化路径转发。

例如：

```text
用户 A → 最近边缘节点 → 优化后的全球中继路径 → 用户 B 附近节点 → 用户 B
```

这部分比“使用 UDP 协议”更接近护城河。

#### 2.5 NAT 穿透

很多用户在家庭路由器、公司网络、移动网络之后，没有公网 IP。RTC 需要让双方尽可能直连，或在无法直连时通过中继服务器连接。

常见机制包括：

- STUN：发现自己的公网映射地址；
- TURN：无法直连时通过中继转发；
- ICE：在多种路径中选择最优连接方式。

这些技术不是独家技术，但在大规模、多地区、复杂网络环境下调优很难。

## 五、UDP 优化有什么护城河？

### 1. UDP 本身没有护城河

UDP 是协议，人人都能用。基础 WebRTC 也不是强护城河，因为开源方案和云厂商都能实现。

真正的护城河在于：

> 围绕 UDP 构建的实时传输系统。

### 2. 护城河来源一：长期积累的网络算法

同样使用 UDP，不同厂商体验可能差很多。差异来自：

- 什么时候重传；
- 什么时候丢弃；
- 冗余包发多少；
- 弱网下先牺牲视频还是音频；
- 码率下降多快；
- 网络恢复后码率提升多快；
- 多人房间里不同流的优先级如何安排；
- AI 语音场景下如何保证打断低延迟。

这些不是写一个 Demo 就能解决的，需要大量真实场景调参。

### 3. 护城河来源二：全球实时网络节点

如果只是点对点 UDP，护城河有限。若叠加全球边缘节点、跨区域中继网络、实时路径选择，才会形成基础设施壁垒。

尤其在跨国社交、海外直播、弱网地区、全球 AI Agent 服务等场景中，节点和路由调度能力非常关键。

### 4. 护城河来源三：海量真实网络数据

RTC 系统使用量越大，越能知道：

- 哪些地区网络质量差；
- 哪些运营商在晚高峰拥堵；
- 哪些链路容易丢包；
- 哪些手机型号兼容性差；
- Wi-Fi 和 5G 切换时如何处理；
- 不同国家的网络监管、防火墙、运营商环境如何影响连接。

这些数据会反过来优化路由和算法，形成经验壁垒。

### 5. 护城河来源四：客户端 SDK 工程能力

RTC 不只是服务器问题，也强依赖终端。

要适配：

- Android 各种机型；
- iOS；
- Windows；
- macOS；
- Web 浏览器；
- 小程序；
- Unity、Unreal 游戏引擎；
- IoT 和智能硬件。

还要处理麦克风权限、蓝牙耳机、回声消除、噪声抑制、摄像头采集、硬件编码、前后台切换、省电和发热等问题。

普通开发者能调通 Demo，但很难保证全球几千万用户、几千种设备稳定运行。

### 6. 护城河来源五：实时运维能力

高端 RTC 客户不是只买 SDK，还需要 SLA、监控、问题定位和故障诊断。

例如客户会问：

- 昨晚印尼用户为什么卡顿？
- 某个直播间为什么连麦失败？
- 某款 Android 机型为什么回声严重？
- AI 客服延迟高，是模型问题、ASR 问题、TTS 问题、RTC 网络问题，还是客户端采集链路问题？

RTC 厂商需要提供实时质量监控、日志、告警、诊断和自动调度。这对大客户很重要。

## 六、声网技术中最有护城河的是什么？

### 1. 最强护城河：SD-RTN 全球实时传输网络

声网最有护城河的技术资产是 SD-RTN 全球实时网络。

它的价值不是“传输数据”，而是：

- 在复杂公网环境下选择更优路径；
- 在跨国、弱网、移动网络中降低延迟和丢包；
- 在多人互动和高并发场景中保持稳定；
- 在故障时能快速切换、诊断和恢复。

这类能力需要节点建设、路由调度、真实网络数据和运维体系长期积累。

### 2. 第二层护城河：弱网音视频算法

弱网算法包括 FEC、选择性重传、丢包隐藏、抖动缓冲、拥塞控制、自适应码率、音频优先、回声消除、噪声抑制等。

这些算法单独看不是秘密，但组合调优很难。真正的壁垒在于：在各种真实网络、设备、场景下，让体验尽量稳定。

### 3. 第三层护城河：多端 SDK 和设备适配

声网 SDK 覆盖移动端、Web、桌面端、游戏引擎、跨平台框架和 IoT 设备。

这部分很碎，但对体验影响很大。客户看到的是一个 SDK，背后是大量设备、系统、浏览器、音频设备、摄像头、权限、硬编硬解、功耗和发热的兼容性工作。

### 4. 第四层护城河：质量监控与故障诊断

高端客户需要知道问题来自哪里：地区、运营商、链路、设备、SDK、模型接口、ASR、TTS，还是业务侧。

实时质量监控和故障诊断能力会提高客户粘性和替换成本。

### 5. AI 语音是新机会，但还不是单独最强护城河

声网的 Conversational AI Engine 符合需求演变，但 AI 编排层本身竞争激烈。LiveKit、Twilio、大型云厂商都在进入。

因此，声网在 AI 中的真正护城河不是“接入 LLM”，而是：

> 当 AI Agent 需要进入真实语音环境、弱网地区、嘈杂空间、智能硬件、多端设备和大规模并发时，声网原有 RTC 网络和音频工程能力能否显著降低延迟和失败率。

## 七、护城河强弱排序

| 技术 / 能力 | 护城河强度 | 判断 |
|---|---:|---|
| SD-RTN 全球实时网络 | 最强 | 节点、路由、实时调度、数据积累、运维体系难复制 |
| 弱网音视频算法 | 强 | 需要长期真实场景调优，不是单一算法 |
| 多端 SDK 兼容性 | 强 | 移动端、Web、游戏引擎、IoT 适配复杂 |
| QoE 监控与诊断 | 中强 | 大客户粘性来源，提升替换成本 |
| Conversational AI Engine | 中强，待验证 | 顺趋势，但竞争者也多 |
| 普通视频 / 语音 SDK | 中等偏弱 | WebRTC、开源 SFU、云厂商都能做 |
| UDP 优化本身 | 弱 | UDP 是协议，不是护城河 |

## 八、投资研究中的关键验证指标

判断声网技术护城河是否真正转化为商业价值，需要跟踪以下指标：

1. **净收入留存率是否超过 100%**  
   如果高端客户持续扩容，NRR 应该改善并超过 100%。

2. **AI 相关收入占比和增长速度**  
   需要验证 Conversational AI Engine 是否真正带来增量，而不仅是概念。

3. **毛利率是否企稳**  
   AI 语音可能增加算力、带宽和模型调用成本，需要观察是否压低毛利率。

4. **大客户数量和行业分布**  
   如果客户从娱乐社交扩展到金融、教育、医疗、智能硬件、企业服务，说明技术能力的可迁移性更强。

5. **非中国市场增长情况**  
   全球实时网络的价值，需要通过海外客户增长来验证。

6. **高端场景占比**  
   若增长来自普通音视频，护城河较弱；若增长来自 AI 语音、游戏语音、跨境互动、智能硬件，则护城河价值更高。

## 九、最终判断

声网技术中最有护城河的不是 UDP，也不是普通音视频 SDK，而是：

> 全球实时传输网络、弱网音视频算法、多端 SDK 适配、实时质量监控与故障诊断共同构成的实时体验保障系统。

这套能力在普通 RTC 场景中会被商品化，但在以下场景中仍然有较强价值：

- 游戏语音；
- 语聊房；
- 直播连麦；
- 跨国社交；
- 在线教育；
- 金融视频客服；
- 医疗远程互动；
- 智能硬件；
- AI 语音 Agent；
- AI Avatar。

因此，对声网的核心判断应是：

> 未来增量需求是否越来越偏向低延迟、弱网、多端、高并发、AI 语音这些高难场景。

如果答案是肯定的，声网的技术护城河仍有价值；如果需求主要停留在普通音视频通话，声网的护城河会被 WebRTC、开源方案和云厂商持续压薄。
